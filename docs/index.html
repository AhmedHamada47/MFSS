Why MFSS

Enterprise-Grade Features

Built for production migrations with resilience, observability, and safety at every step.

⚡

Parallel Transfers

Configurable concurrency up to 100 parallel downloads with token-bucket rate limiting for precise throughput control.

🔄

Resumable by Design

Upsert-based logging means re-running is always safe. Only pending records with retries remaining are processed.

💣

Circuit Breaker

After N consecutive failures, the circuit opens to prevent cascading failures. Automatically recovers after a timeout.

🔁

Full Rollback

One command to delete all successfully migrated files. Dry-run supported for rollback preview.

📈

Real-Time Progress

Live progress bar with throughput, counts, and ETA. Toggle verbose mode for per-file details.

💡

Dry-Run Mode

Preview exactly what would happen without touching any files or databases. Validate your config safely.

🔎

Pre-Flight Validation

FluentValidation checks every setting, connection string, and provider field before any operation begins.

🔒

Secret-Safe by Default

Use ${ENV_VAR} placeholders for credentials. Connection strings are masked in logs. No secrets in config files.

🔃

URL Change Detection

If a source URL changes for a previously migrated file, it's automatically reset for re-migration.

〈 Workflow A: Install as CLI Global Tool

Best for: Automation, CI/CD pipelines, server environments, and developers who live in the terminal.

1

Install the .NET SDK

Download and install the .NET 9 SDK from dotnet.microsoft.com. Verify it's installed:

dotnet --version
bash
2

Install MFSS as a global tool

NuGet will download and install the latest version from nuget.org:

dotnet tool install --global MFSS
bash

To update later: dotnet tool update --global MFSS

3

Verify installation

The tool is now available as the mfss command in your terminal:

mfss --version
mfss --help
bash
4

Create your configuration file

Create appsettings.json with your source DB, storage providers, and migration settings:

{
  "Migration": { "Name": "my-migration", "ParallelDownloads": 4 },
  "SourceDb": {
    "ConnectionString": "Server=localhost;Database=source;Integrated Security=true;",
    "Tables": [{ "TableName": "MediaFiles", "UrlColumn": "Url" }]
  },
  "SourceFileSystem": { "Type": "local", "BasePath": "/data/files" },
  "DestinationFileSystem": {
    "Type": "s3",
    "BucketName": "my-bucket",
    "Endpoint": "https://s3.amazonaws.com",
    "AccessKey": "${AWS_ACCESS_KEY}",
    "SecretKey": "${AWS_SECRET_KEY}"
  },
  "DestinationDb": { "ConnectionString": "Server=localhost;Database=migration_log;Integrated Security=true;" }
}
json
5

Run a dry-run to validate

Preview the migration without touching any files or databases:

mfss --dry-run --mode migrate --config appsettings.json
bash

Add --verbose to see every file that would be transferred.

6

Execute the migration

Run the actual migration. Files will be transferred in parallel with progress tracking:

mfss --mode migrate --config appsettings.json
bash
7

Rollback if needed

If something goes wrong, rollback all migrated files:

# Preview rollback first
mfss --dry-run --mode rollback --config appsettings.json

# Execute rollback
mfss --mode rollback --config appsettings.json
bash

〈 Workflow B: Use via VS Code / NuGet Package Manager

Best for: .NET developers who want to integrate MFSS as a project dependency, build custom tooling on top of it, or prefer GUI package management.

1

Open your project in VS Code

Open your .NET project (or create a new one) where you want to reference the MFSS package.

2

Open NuGet Package Manager

In VS Code, press Ctrl+Shift+P to open the command palette, then type NuGet Package Manager and select it. Or right-click the Dependencies node in the Solution Explorer and choose Manage NuGet Packages.

3

Search for "MFSS"

In the NuGet Package Manager, click Browse tab and search for MFSS. Select it from the list.

Alternative: Install via the .NET CLI by adding a package reference to your project: 

dotnet add package MFSS --version 1.0.2
bash
4

Install the package

Click the Install button next to the MFSS package. VS Code will add a package reference to your .csproj file:

<!-- Added to your .csproj file -->
<PackageReference Include="MFSS" Version="1.0.2" />
xml
5

Run MFSS from your project

If installed as a tool reference, run with dotnet tool run mfss. If installed as a library reference, use the MFSS classes and services in your own code:

// Use MFSS services in your own .NET application
using MFSS.Services;
using MFSS.Models;

var config = new MigrationSettings { Name = "my-migration", ParallelDownloads = 4 };
var orchestrator = new MigrationOrchestrator(...);
var exitCode = await orchestrator.RunAsync();
csharp
6

Create and run a migration

Write a migration script or use the built-in CLI by calling the MFSS entry point from your project's build output:

# Run via dotnet tool (if installed as local tool)
dotnet tool run mfss -- --mode migrate --config appsettings.json

# Or run the project directly
dotnet run -- --mode migrate --config appsettings.json
bash
💡 Which workflow should I choose? Use the CLI Global Tool for running migrations directly from the terminal, automation scripts, and CI/CD pipelines. Use the NuGet Package if you want to build custom applications on top of MFSS, extend its functionality, or manage versions through your IDE's package manager.

Real World

Examples

Common migration patterns to get you started quickly.

☁ Local Files to Cloudflare R2

{
  "Migration": { "Name": "media-to-r2", "ParallelDownloads": 8, "RateLimitPerSecond": 50 },
  "SourceDb": {
    "ConnectionString": "Server=localhost;Database=media;Integrated Security=true;",
    "Tables": [{ "TableName": "MediaFiles", "UrlColumn": "FilePath", "IdColumn": "MediaId" }]
  },
  "SourceFileSystem": { "Type": "local", "BasePath": "C:\\media\\files" },
  "DestinationFileSystem": {
    "Type": "s3",
    "BucketName": "my-bucket",
    "Endpoint": "https://<id>.r2.cloudflarestorage.com",
    "Region": "auto",
    "AccessKey": "${R2_ACCESS_KEY}",
    "SecretKey": "${R2_SECRET_KEY}"
  },
  "DestinationDb": { "ConnectionString": "...migration_logs..." }
}
json

🔲 Azure Blob to GCS with Third DB Update

{
  "SourceFileSystem": {
    "Type": "azure",
    "AzureConnectionString": "${AZURE_STORAGE_CONN}",
    "ContainerName": "media-assets"
  },
  "DestinationFileSystem": {
    "Type": "gcs",
    "GcsBucket": "my-gcs-bucket",
    "GcsCredentialPath": "${GCS_SA_JSON}"
  },
  "ThirdDb": {
    "Enabled": true,
    "ConnectionString": "Server=db;Database=app;...",
    "UpdateQuery": "UPDATE assets SET file_url = {url} WHERE id = {id}"
  }
}
json
ℹ Third DB Transactions All URL updates run inside a SQL transaction. If any update fails, everything is rolled back — all or nothing.
How It Works

Under the Hood

The orchestrator workflow that powers every migration.

1 Load & Validate Config
→
2 Create Log Tables
→
3 Fetch Source Records
→
4 Register in Log
→
5 Transfer Files
→
6 Update Third DB
→
7 Summary & Exit

For rollback mode, steps 3-5 are replaced with: Query success records → Delete from destination.
